Rails 程式最佳實務

Good code is its own best documentation. As you’re about to add a comment, ask yourself, ‘How can I improve the code so that this comment isn’t needed?’ Improve the code and then document it to make it even clearer. – Steve McConnell, Code Complete 作者

到目前為止我們主要學習如何讓程式完成我們想要的功能，而真正專業的程式設計師精益求精，思考如何讓程式寫的漂亮、容易維護，打造高品質的軟體工藝。這一章我們來討論這個主題。當然，在修改重構程式前，最好擁有自動化測試，以確保程式於修改後執行無誤。

基本觀念

隨著日漸複雜的程式和多人團隊開發的演進，你的程式會變得：

  • 僵硬 (Rigidity)：難以修改，每改一處牽一髮動全身
  • 脆弱 (Fragility)：一旦修改，別的無關地方也炸到
  • 固定 (Immobility)：難以分解，讓程式再重用
  • 黏滯 (Viscosity)：彈性不夠，把事情做對比做錯還難
  • 不需要的複雜度 (Needless Complexity)：過度設計沒直接好處的基礎設施
  • 不需要的重複 (Needless Repetition)：相同概念的程式碼被複製貼上重複使用
  • 晦澀 (Opacity)：難以閱讀，無法了解意圖

我們希望的好程式有以下特性：

  • Readability 易讀，容易了解
  • Flexibility 彈性，容易擴充
  • Effective 效率，撰碼快速
  • Maintainability 維護性，容易找到問題
  • Consistency 一致性，循慣例無需死背
  • Testability 可測性，元件獨立容易測試

具體的說，在MVC的框架中，我們希望可以更細節組織程式碼。
